How it works

Greenroom's deal engine now handles Vs and %-of-net, recoups with structured waterfall positions, tier ratchets, and an expense cap. Structured deal entry closes both gaps.

1. Supported deal types

Now supported by the engine

flat guarantee only, no expense deduction from artist

percentage_of_gross artist % of adjusted gross

percentage_of_net artist % of net after expenses

vs greater of guarantee or % side — ties go to %

Vs honors percentageBasis (gross or net) on the deal.

Not supported

door

Returns { supported: false, kind: "deal_type_unsupported" }. Door deals need split door receipts & presales that ticketing data doesn't carry yet.

2. Mix & match — what plugs into what

Recoups, bonuses, and ratchets are all orthogonal to the base deal type. Any combination is legal on a supported deal.

3. Recoup data model — two coupled JSON blobs

Recoups live in two places: a RecoupClause on the deal (negotiated up front) and a Recoup on the settlement (the actual billed amount). The settlement-time row references the clause by id.

A. Deal-time clause — deals.recoupClausesJson

Negotiated rights agreed up front. Array of RecoupClause.

[ { id: string, label: string, category: …, waterfallPosition: …, amountModel: …, fixedAmount: number | null, capAmount: number | null, overageAbsorbedBy: …, overageSplitPct: number | null, rationale: string | null } ]
fieldvalueswhy
category marketing, prior_advance, production_overage, hospitality_overage, backline, damages, other Categories the Coastal Spell thread actually argues about.
waterfallPosition off_gross, inside_expense_cap, outside_cap_pre_split, post_split The $900 marketing recoup question — gross or in the $2,500 expense bucket — has a structured answer now.
amountModel fixed, actual_up_to, actual_uncapped "$500 marketing" vs. "actuals up to $2,500" vs. "whatever it costs" are three different deals.
fixedAmount number, required when amountModel = fixed The agreed dollar number.
capAmount number, required when amountModel = actual_up_to Hard ceiling on actuals.
overageAbsorbedBy venue, artist, split, n_a Who eats it when actuals blow past the cap.
overageSplitPct 0–1, required when overageAbsorbedBy = split Artist's share of the overage in a split.
rationale free text Booker note — surfaces during dispute review.

B. Settlement-time application — settlements.recoupsJson

Actual billed amounts at settlement. Array of Recoup. Joins back to clause by id.

[ { id: string, // matches RecoupClause.id category: …, label: string, amount: number, // actual billed status: … } ]
fieldvalueswhy
id matches RecoupClause.id, or standalone Standalone is the legacy fallback — treated as outside_cap_pre_split.
category marketing, hospitality_overage, production_overage, prior_advance, backline, damages, other Same enum as the clause; unknown → other.
amount number (dollars) What the venue is actually billing.
status agreed, disputed, withdrawn Recoups have their own lifecycle — can be disputed after the rest is signed. withdrawn skips the line entirely.

4. How the engine applies recoups — waterfall flow

Every supported deal type runs the same waterfall. Each clause's waterfallPosition picks which bucket it falls into. Flow reads left-to-right, then drops to the next row.

1
Gross box office  tickets + counted comps
↓
2
− off-gross recoups  off_gross clauses come off gross before fees
↓
3
− fees  = net box office
↓
4
− expenses + inside-cap recoups  inside_expense_cap clauses share the expenseCap ceiling with regular expenses
↓
5
− outside-cap recoups  outside_cap_pre_split clauses come off net before the artist split
↓
6
Artist split  flat / %-of-gross / %-of-net / vs (winner) — applied to the result of step 5
↓
7
− post-split recoups  post_split clauses + any cap overages absorbed by the artist
↓
8
Total to artist
Amount resolution per clause — fixed → use fixedAmount. actual_up_to → min(billed, capAmount), route overage by overageAbsorbedBy. actual_uncapped → use full billed amount. withdrawn applications skip entirely.

5. Tier ratchets — partially supported, with a known sharp edge

What works

net-dollar bands

e.g. "60% of net to $4,000, then 70% above" — payout is computed band-by-band, each band gets its own line in the worksheet.

capacity-fraction tiers

e.g. "80% to 100% sold @ 85%" — uses tickets / capacity, picks the single matching tier.

Validations enforced

  • tier 1 must start at 0
  • tiers must be sorted by from
  • no gaps and no overlaps between bands
  • only the last tier may be open-ended (to: null)
  • to must be greater than from

Violations return deal_misconfigured with a fixable reason.

Known sharp edge — basis inference

The schema does not distinguish net-dollar tiers from capacity-fraction tiers. The engine guesses with a heuristic:

if any from/to is > 0 and < 1 → capacity_fraction else → net_dollars

Works because seed data has a gap (fractions ≤ 1, dollars ≥ 4000) but a deal with a $0.50 band would mis-classify.

Fix queued: add an explicit basis: "net_dollars" | "capacity_fraction" discriminator on the ratchet bonus. See lib/dealMath.ts:521.

6. What structured deal entry closes

Deal types problem — closed

Vs and %-of-net are first-class on the engine, not just the schema. The 82% of customers going around the product, and the 53% using it only for term definitions, can do the math here now.

Ambiguity problem — closed for the structured path

The Coastal Spell marketing recoup has a single structured answer: waterfallPosition says where it lands, amountModel + capAmount say how much. The $2,500 expense bucket question is no longer free-text vs. free-text.
